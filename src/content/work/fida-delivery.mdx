## The opening

Bishoftu is a city of two million people 47 km southeast of Addis Ababa, with no reliable on-demand delivery network. Local commerce runs on phone calls, motorbike couriers paid in cash, and a wall of *"can you bring this to my office?"* WhatsApp threads. FIDA's bet is that a single, well-designed app pair — customer and rider — can replace that informal layer with something verifiable, dispatchable, and economically sustainable.

I co-founded FIDA and built the design and frontend end-to-end. The product had to launch in a market with low smartphone uniformity, weak data connections, and zero brand recognition. The apps had to be obviously trustworthy on the first screen, fast on 3G, and small enough to install over patchy Wi-Fi.

## Constraints that shaped the build

- **Two apps, one operator.** A unified design system was not a nice-to-have — it was the only way to ship both apps before runway pressure hit.
- **Multilingual from day one.** Amharic and English need first-class treatment in every screen — including locale-aware numerals and date/time formatting via `i18next`.
- **Offline-resilient.** Riders lose signal in the city's older neighborhoods; orders cannot be lost when the radio drops.
- **Trust signals over feature breadth.** First-time users need to see verified riders, clear pricing, and a real phone number before they will tap *Order*.

## How it's built

The stack settled on **Expo SDK 56** with the **New Architecture** turned on, **continuous native generation** so we never commit `ios/` and `android/`, and **EAS Build + Updates** as the release pipeline. Server state is **TanStack React Query** with **persist + sync storage** so the apps cold-start instantly with the last known shape of orders, riders, and pricing. Local persistence uses **react-native-mmkv** (sync, fast) instead of AsyncStorage.

Maps render through **`@maplibre/maplibre-react-native`** on both customer and rider sides — the same vector tile source the FIDA marketing site uses, which keeps coverage logic and styling consistent across web and mobile. Push and in-app notifications run through **Notifee** + **React Native Firebase Messaging**. Animations are **Reanimated 4** (worklets for any non-trivial gesture). Forms use **React Hook Form**. Icons come through **`lucide-react-native`**.

A shared `@fida/ui` package owns the design tokens, the typography scale, and a primitive set — Button, Sheet, Field, Badge, Pill, MapView, AddressCard — that both apps consume identically. Color, radius, and motion are bound to tokens so a single change in the theme file updates both apps and the marketing site.

A `MockServer` adapter sits between the apps and `https://api.fidadelivery.et/api/v1`. Default-on mock data means every screen renders without the backend — designers can iterate without waiting on infra, and QA can reproduce any state by toggling a fixture.

## What ships today

- **Customer app**: address search, delivery price quote, MapLibre rider tracking, in-app calling, ride history, profile, push notifications.
- **Rider app**: shift control, foreground location during shift only, order acceptance, navigation handoff, SOS-to-ops, daily earnings view.
- **OTA channel** wired on customer (preview + production); rider on the same pipeline by next release.

## What I learned

Vertical integration — same designer, same engineer across two apps and a website — let the brand cohere in a way a typical contractor handoff never produces. A pill in the customer app feels exactly like a pill in the rider app, because it *is* the same pill. The cost is that the bus factor is one. The next phase is hiring a second engineer onto the shared component package so the system survives me.
